# 2026-05-15 M3 灰化决策

> 上级 spec: `docs/specs/m3-yt-video-detail.md`(已写完但**搁置**)
> 决定: M3 详情页代码全部保留,但**入口禁用**(YT 卡片 onTap 改回 Toast 占位),等 M3v2 重新设计后启用

## 时间线

1. 真机测 1: 点 YT 卡片 → **全屏灰色**(无信息)
2. 注入 `ErrorWidget.builder` 红屏诊断 → 真机测 2: `type 'Null' is not a subtype of type 'VideoType'` at `controller.dart:415`
3. 修 1: YT 卡片 args 加 `videoType: VideoType.ugc` → 真机测 3: `type 'Null' is not a subtype of type 'String'` at `controller.dart:424`
4. 修 2: YT 卡片 args 加 `bvid: ''` → 真机测 4: **完全卡死**,没红屏也无 toast 反馈
5. 注入 6 个追踪 toast(VD.onInit / initYtSource start / detail ok / muxedUrl / setDataSource pre/done / CATCH)→ 真机测 5: **依然完全卡死,一个 toast 都没冒出来**
6. 用户拍板: M3 灰化,M3v2 重新设计

## Root cause(已识别但不修)

`VideoDetailController` 实质是**重耦合 B 站特定字段**的 controller(`late String bvid`、`late int aid`、`isFileSource` 范式 27 处引用、heroTag 链接 dimension/cid 等)。

M3 spec 当时的选择 "沿用 isFileSource 范式" 看似省力,**实际上把 YT 塞进了一个不是为它设计的容器**。每次进详情页都会让 onInit 里一连串 cast 引发 null cast,即便补完字段也只能让 onInit 通过 — **`navigation/hero 动画 + initState 期间整个详情页 layout 树 build 失败**(具体在 VD.onInit 之前已挂,toast 都未触发)。

复杂度被显著低估。

## 灰化范围(本次实际改动)

```
M lib/main.dart                                # 删 ErrorWidget.builder 诊断
M lib/pages/video/controller.dart              # 删 6 个 trace toast
M lib/common/widgets/video_card/yt_video_card_h.dart  # onTap 路由跳转 → Toast 占位
```

**保留的 M3 代码**(M3v2 时直接复用):
```
A lib/services/youtube/yt_video.dart            # YT 元数据 / 取流 service
A lib/models_new/youtube/yt_video_detail.dart   # YT 详情 model
A lib/pages/video/introduction/yt/controller.dart  # YT IntroController
A lib/pages/video/introduction/yt/view.dart        # YT 简介 panel
M lib/models/common/video/source_type.dart      # SourceType.youtube 枚举(暂无路由产生)
M lib/pages/video/controller.dart               # isYtSource / initYtSource / ytVideoDetail / ytMuxedUrl 字段及方法
M lib/pages/video/view.dart                     # isYtSource 分发分支
```

**逻辑等价性**: 任何 `SourceType.youtube` 路由都没了入口 → `isYtSource` 永远 false → `(isFileSource || isYtSource)` 永远等于 `isFileSource` → **B 站行为零回归**。

## 用户体验影响

- ✅ 搜索混排正常(M2 全部能力保留)
- ✅ YT 卡片缩略图、标题、作者、时长、播放量正常
- ✅ 点 YT 卡片 → Toast `YouTube 详情页暂未启用`(不再卡死)
- ❌ YT 视频暂时无法播放
- ✅ B 站详情页 / 评论 / 弹幕 / 相关视频 全部零回归

## M3v2 设计方向(不在本次,留作下次任务)

**核心**: 不复用 `/videoV` 路由,**不复用 `VideoDetailController`**,新建独立轻量 page。

```
新建 lib/router/yt_video_page.dart  # GetPage('/ytVideo', ...)
新建 lib/pages/yt_video/controller.dart  # 独立 GetxController,只承载 YT 字段
新建 lib/pages/yt_video/view.dart        # 独立 Scaffold,只组装: 播放器 + YT 简介
复用 PlPlayerController                  # 播放器 widget 独立性强,可复用
复用 lib/services/youtube/yt_video.dart  # 数据通路本次已写好
复用 lib/pages/video/introduction/yt/*   # YT 简介 widget 本次已写好(可能要调整 controller find 方式)
```

复用率约 60%(数据通路 + 简介 widget 都现成),只重写"详情页骨架 controller + view"和路由。

## 教训归档

- **"复用现有路由 + 控制器"看似省事,实际可能掉进强耦合陷阱**。下次接新源前,先 grep 目标 controller 的 `late` field 数量;高的就别复用,该独立时独立。
- **诊断手段升级路径有效**: 灰屏 → ErrorWidget 红屏 → 6 toast。三步把范围一路收缩,**避免凭空猜测**。这条路径在 memory `feedback_no_local_deadloop.md` 实践了一次。
- **MVP 三角**(范围 / 时间 / 质量)— 当一个里程碑被低估后,**灰化 + 留 building block** 比"硬修到通"或"完整回退"都好,前者保用户体验,后者保未来工作不浪费。
